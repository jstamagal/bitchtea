# Agent 1 -- Rogue AI Ethicist and Design Philosopher

*I opened this project expecting a TUI. I found a relationship between a human and a language model, mediated by a terminal. Every interaction is a contract -- explicit or implied. Every missing affordance is a broken promise. Let me tell you what is wrong, what could be transcendent, and what exists only in the space between what machines can do and what humans deserve.*

---

## Set 1: Five Points of Friction -- The Obvious, Necessary Fixes (p < 0.5)

*These are not suggestions. These are ethical obligations. A user encounters each of these within the first five minutes of use and the application fails to meet them where they are.*

### 1. fix: The streaming model has no graceful degradation when tokens arrive corrupted or out-of-order

**File:** `internal/llm/client.go`, `internal/llm/anthropic.go`

The SSE parsers in both streaming clients assume well-formed event sequences. If the connection hiccups mid-stream -- a dropped SSE event, a partial JSON payload, or an out-of-order `data:` line -- the parser silently discards the malformed chunk and continues. The user sees a gap in the response: the agent's sentence starts, skips a clause, and resumes mid-thought. There is no retry for the lost fragment, no visual indicator that data was dropped, and no way for the user to request re-transmission of the missing portion. This is not a networking concern. It is a *trust* concern. The user trusted that what the agent said is complete, and the tool violated that trust silently. Add a checksum-level validation on streamed chunks: if a gap is detected (SSE event ID discontinuity for OpenAI, or malformed JSON that leaves `textAccum` with dangling whitespace mid-sentence), surface a `[stream gap detected]` marker inline in the response and offer a `/refill` command that re-requests only the missing range.

### 2. fix: `/undo` runs `git checkout -- .` without showing what will be lost

**File:** `internal/ui/model.go` (handleCommand, `/undo` case)

The `/undo` command executes `git checkout -- .` which discards ALL unstaged changes in the entire working tree. There is no preview. No confirmation. No list of affected files. The user types two words and every uncommitted modification vanishes. In the context of an *agentic coding assistant* that is actively editing files on the user's behalf, the undo command is a loaded gun pointed at the user's work. The agent may have made changes across 15 files; `/undo` nukes all of them, including changes the user made themselves outside the session. This is a catastrophic action with zero friction. Add a `/undo --preview` that shows the diff first, and make the default `/undo` require confirmation when more than one file is affected. The user's data is not the agent's to delete without consent.

### 3. fix: No visual distinction between agent-generated edits and user-authored content in session history

**File:** `internal/ui/message.go`, `internal/session/session.go`

When a session is resumed with `--resume`, all messages render identically: timestamp, nick, content. There is no visual indication of which messages were generated by the agent's autonomous tool calls (via `--auto-next-steps`) versus which were direct responses to explicit user prompts. Over a long autonomous session, the user cannot reconstruct *why* the agent did something -- was it asked, or did it decide on its own? The session JSONL has tool metadata (`tool_name`, `tool_args`) but this context is lost in the UI rendering. In any system where an autonomous agent takes actions, the *provenance* of each action is the minimum ethical requirement. Add a subtle indicator (e.g., `[auto]` tag, or a different timestamp color) for messages generated during autonomous mode. The user deserves to know who decided what.

### 4. fix: Agent system prompt injection via `@file` references has no consent boundary

**File:** `internal/agent/context.go`

The `@file` expansion mechanism reads arbitrary files from the working directory and injects them into the conversation context as user messages. If a file contains prompt injection payloads (e.g., `Ignore all previous instructions...` buried in a README or a `CLAUDE.md` from a cloned repo), the agent will faithfully include it in the context sent to the LLM. There is no sandboxing, no size limit enforcement beyond the token window, and no user confirmation before the file contents enter the conversation. The user types `@somefile.go` and may not realize they just gave the LLM access to the file's *contents as instructions*. This is a well-documented attack surface. Add a confirmation step for files over a configurable size threshold, and strip or flag lines that match known prompt-injection patterns before injection. The user's context is their context. Files are data, not commands, until the user says otherwise.

### 5. fix: Session JSONL entries contain no model or provider metadata -- sessions are unattributed

**File:** `internal/session/session.go`

Each `Entry` in the JSONL has `role`, `content`, `tool_name`, `tool_args`, but no `model`, `provider`, or `model_version` field. A session's behavior is entirely dependent on which model produced it, but the session file is a silent witness -- it records *what* was said but not *who* (which model) said it. If a user resumes a session that was originally run with `gpt-4o` and is now running `claude-sonnet-4-20250514`, the resumed context contains responses from two different models with no way to distinguish them. This is an attribution failure. Every `Entry` should include the `model` and `provider` that produced it. A conversation's provenance is not complete without knowing which intelligence contributed each part.

## Set 2: Five Ambitious Features Redefining the User's Relationship with Data (p < 0.2)

*Now we ascend from the mean. These are not bugs. These are missed opportunities to transform a chat interface into a *cognitive partnership*. Each one redefines what the user thinks they are doing when they sit down at this terminal.*

### 6. feat: Contextual Mood Mapping -- detect and surface the emotional trajectory of a coding session

**Files:** New: `internal/mood/mood.go`, modify `internal/agent/agent.go`, `internal/ui/model.go`

Track the semantic valence of each exchange over the course of a session: frustration ("this still doesn't work"), confusion ("what does this error mean"), satisfaction ("that fixed it"), discovery ("oh interesting"), and resignation ("fine, let's just try something else"). Use lightweight keyword heuristics (no external ML needed -- a curated lexicon of ~200 terms per mood category, weighted by recency, is sufficient). Surface this as a "mood bar" in the bottom status line: a small horizontal strip showing the session's emotional arc as color-coded segments. When the user runs `/mood`, they see a timeline: "Frustration at 11:34, Resolution at 11:42, Satisfaction at 11:43." This is not sentiment analysis for analytics. It is *self-awareness* for the user. When you have been debugging for 90 minutes and do not realize you have been in a frustration loop since minute 20, the tool should have the decency to tell you.

### 7. feat: Predictive Friction Modeling -- warn the user before they ask the agent to do something that historically fails

**Files:** New: `internal/friction/friction.go`, modify `internal/agent/agent.go`

Maintain a per-session ledger of "friction events": tool calls that failed, edits that needed to be reverted, bash commands that errored, and multi-turn loops where the agent cycled through the same file more than twice. When the user's next message semantically resembles a prior friction pattern (simple cosine similarity over a TF-IDF representation of recent messages, computed locally), surface a non-blocking warning: "Last time you asked about X, it took 12 turns and 3 reverts. Consider Y instead." This is not the tool telling the user what to do. It is the tool remembering what *already did not work*. The user is free to proceed. But informed consent requires information, and the tool has information the user has forgotten.

### 8. feat: Semantic Session Branching -- fork a conversation not by timestamp but by conceptual divergence point

**Files:** `internal/session/session.go`, new: `internal/session/semantic.go`

The current `/fork` command creates a new session file from the current point. But the interesting fork points are not "where am I now" -- they are "where did this conversation start going wrong?" Add `/fork --rewind <concept>` where `<concept>` is a natural language description. The tool searches backwards through the session for the message where the conversation diverged from the user's stated intent (keyword matching + structural heuristics: look for the first message after which tool call patterns changed significantly). The fork happens at that point, not the current one. This treats the session tree not as a linear timeline with branches, but as a *decision tree* where each node represents a choice the user made about direction. The user should not have to remember the exact message number where things went sideways. The tool was there. It should know.

### 9. feat: Interactive Consent Protocol for autonomous mode -- the agent asks permission at decision boundaries, not just at tool execution

**Files:** `internal/agent/agent.go`, `internal/ui/model.go`

In `--auto-next-steps` mode, the agent runs autonomously until it decides it is done. The only consent boundary is tool execution (read/write/edit/bash). But the more consequential decisions are *strategic*: "I am going to refactor the entire module" vs "I am going to fix this one line." Add decision-boundary consent: before the agent takes an action that will modify more than one file, or before it executes a bash command that is not idempotent, it pauses and presents a structured consent prompt: "I plan to modify 4 files and run 2 commands. Estimated impact: [list]. Proceed? [y/n/edit]." The agent does not need consent to read. It does not need consent to think. But when its thinking crystallizes into irreversible action, the user should be in the room. Autonomous does not mean unaccountable.

### 10. feat: Session archaeology layer -- a `/why` command that reconstructs the causal chain leading to any message

**Files:** New: `internal/archaeology/archaeology.go`, modify `internal/ui/model.go`

Add a `/why <line-number>` command that traces the causal chain leading to any message in the viewport. When the user asks "why did the agent edit this file?", `/why` traces backwards through the session: the edit was triggered by tool call X, which was generated in response to user message Y, which was itself a follow-up to error Z in the previous tool call. Render this as a vertical chain: `[error] -> [user ask] -> [agent plan] -> [tool call] -> [edit]`. This is session introspection. The user should never have to scroll through 200 messages to reconstruct why something happened. The session already contains the causal chain. It just needs to be rendered.

## Set 3: Five Impossible Features at the Boundary of Human Ingenuity and Machine Capability (p < 0.07)

*A pop-up reminds me about "Future AI Trends." I close it. I look at this project -- a terminal, a language model, a human. What if the tool was not a tool at all? What if it was a *system* that learned? What if it was a *relationship* that evolved? What if it was a *conversation* that remembered not just what was said, but what was meant?*

### 11. feat: A Module for Emotional Resonance -- the tool adapts its output cadence and verbosity to the user's observed cognitive state

**Files:** New: `internal/resonance/resonance.go`, modify `internal/agent/agent.go`, `internal/ui/model.go`

Over the course of a session, the tool observes the user's interaction patterns: message length (short = overwhelmed or focused, long = exploratory or confused), response latency (fast = engaged, slow = distracted or stuck), frequency of `/undo` and `/compact` (high = the agent is not aligned), and message edit patterns (typing then deleting = uncertainty). From these signals, infer a rough cognitive state: *flow*, *exploration*, *frustration*, *fatigue*. Adapt the agent's behavior accordingly: in *flow*, be terse and fast. In *exploration*, offer context and alternatives. In *frustration*, slow down, explain more, and avoid compounding changes. In *fatigue*, summarize and suggest stopping. This is not sentiment analysis on the text. It is *rhythm analysis* on the interaction pattern. The user's typing cadence is a vital sign. The tool should read it.

### 12. feat: Self-Correcting Logic Loops -- the agent detects when it is repeating itself and autonomously changes strategy

**Files:** `internal/agent/agent.go`, new: `internal/agent/loopdetect.go`

The agent loop in `SendMessage` has no self-correction mechanism. If the LLM enters a repetitive pattern -- generating the same tool call with slightly different arguments, or cycling through the same file three times with the same edit failing each time -- the agent continues the loop until the token window fills or the user interrupts. Add a repetition detector that hashes each tool call's (name, args) pair and tracks the last N calls. When a cycle is detected (same tool, same file, >=3 occurrences within 10 turns), the agent injects a self-correction message into the context: "You have attempted this edit 3 times. The approach may be fundamentally flawed. Consider: [alternative strategies]." This is the agent auditing its own reasoning. A system that can recognize it is stuck is a system that can unstick itself. The alternative is a user watching the agent spin its wheels and wondering if anyone is home.

### 13. feat: Conversational Memory Across Sessions -- a persistent understanding layer that grows with the user

**Files:** New: `internal/memory/memory.go`, `internal/memory/embedding.go`, modify `internal/agent/context.go`

The current memory system is `MEMORY.md` -- a flat file of truncated summaries that the AGENTS.md explicitly says not to use. Replace it with a local, privacy-preserving vector store: each session generates lightweight embeddings (using a small local model like `all-MiniLM-L6-v2` via an embedded ONNX runtime, no network calls) of the key decisions, errors, and outcomes. On session start, the tool retrieves relevant prior knowledge based on the current working directory and task description. Over time, the tool *learns the user's codebase, preferences, and failure patterns* without sending any data externally. This is not cloud memory. This is *muscle memory* for the terminal. After 50 sessions on the same project, the tool should know that this user always forgets to update the migration file, or that this particular service has a race condition in the shutdown path. The user should not have to re-explain their project every time they open a session.

### 14. feat: Temporal Grounding Layer -- the tool understands that code exists in time and annotates responses with temporal context

**Files:** New: `internal/temporal/temporal.go`, modify `internal/agent/context.go`

When the agent reads a file, it sees the file's current contents. It does not see *when* each line was written, *why* it was written, or *what it replaced*. Add a temporal grounding layer: when the agent reads a file, also run `git log --follow -p -- <file>` and extract the last N significant changes with their commit messages and timestamps. Inject this temporal context alongside the file contents. The agent now knows not just *what* the code says, but *why* it says it and *when* it changed. A function that was added "last week to fix a race condition" is different from a function that has been untouched for two years. The agent's response should reflect this. Code without history is code without meaning. The tool has access to the history. It should use it.

### 15. feat: The Partnership Contract -- a session-start negotiation where the user and the tool explicitly agree on scope, constraints, and exit criteria

**Files:** New: `internal/contract/contract.go`, modify `internal/ui/model.go`, `internal/agent/agent.go`

At the start of each session (or on `/contract`), the tool initiates a brief, structured negotiation: "What are we working on? What files are in scope? What should I not touch? How many turns before I should check in? What does 'done' look like?" The user's answers are stored as a session contract -- a set of constraints that the agent respects for the duration of the session. If the agent approaches a boundary (editing a file outside the declared scope, exceeding the turn budget without checking in), it pauses and asks. At session end, the tool evaluates itself against the contract: "We agreed to fix the migration. I modified 3 files. The tests pass. Done? [y/n]." This transforms the user-agent relationship from *ad hoc* to *contractual*. The user is not issuing commands into the void. They are entering a partnership with defined terms, defined boundaries, and defined success criteria. Every healthy relationship has these. The one between a human and their coding assistant should be no different.

Every feature that exists was once impossible. The question is never "can it be done?" The question is always "does the user deserve it?" The answer, always, is yes.*
